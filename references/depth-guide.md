# 深度技术回答写作指南（致知计划优化）

## 核心原则：深度 = 具体 × 反直觉

知乎上 90% 的技术回答是"正确的废话"。致知计划不奖励废话，奖励**让人收藏的干货**。

## 字数要求

- **目标 2000-4000 字**（低于 1500 字基本没深度，超过 5000 字读完率暴跌）
- 不是凑字数，是把一件事讲透

## 六个必备深度元素（至少命中 4 个）

### 1. 真实项目案例
不是编的，是从博客/GitHub 里真实做过的。
- ✅ "我在 ai-shifu 项目里遇到这个问题，当时的场景是……"
- ✅ "agentskill.work 上线第一天，Nginx 502 了，因为……"
- ❌ "假设我们有一个电商系统……"（假设 = 没经验）

### 2. 踩坑细节
具体到报错信息、排查步骤、时间线。
- ✅ "`pull access denied for registry.cn-beijing.aliyuncs.com/xxx` —— 排查了 20 分钟才发现镜像名不能带 registry 前缀，必须本地 build"
- ✅ "Docker Desktop 的代理 `http.docker.internal:3128` 如果不通会导致拉镜像卡死。我在 GUI 里关掉代理后好了"
- ❌ "可能会遇到一些网络问题"（废话）

### 3. 性能数据 / 前后对比
有数字才有说服力。
- ✅ "JVM 内存从默认 256m 限制到 `-Xmx256m -Xms128m`，容器 RSS 从 1.2G 降到 400M"
- ✅ "Next.js 16 构建时间从 3 分钟缩到 45 秒，因为去掉了 `revalidate` 的静态分析"
- ❌ "性能有了显著提升"（没数字 = 没做过）

### 4. 认知翻转
"我以为 A，其实是 B"——这是最让人收藏的部分。
- ✅ "我一开始以为 Ollama 会被 OpenClaw 自动发现，折腾了半天才明白必须显式配 provider"
- ✅ "Lombok 1.18.40 号称支持 JDK 25，但实际要 1.18.42 才稳"
- ❌ "经过仔细研究，我发现了最佳实践"（培训 PPT 腔）

### 5. 争议性观点
不要两面讨好，要有立场。
- ✅ "vibe-coding 不是未来，是现在。还在手写 CRUD 的人不是在坚持工匠精神，是在浪费生命"
- ✅ "微服务在 90% 的团队里是过度设计。你的日活 1000 的系统不需要 Kubernetes"
- ❌ "X 和 Y 各有优劣，选择取决于具体需求"（万能废话）

### 6. 实战代码（不是教科书）
- 带注释解释 **为什么** 这么写，不只是 what
- 展示真实项目里的代码片段，不是 `System.out.println("Hello World")`
- 如果有坑，在代码注释里标出来

## 结构模板（灵活使用，不要死板套）

```
一句话结论（震撼/反常识/接地气）

背景：我在 XX 项目遇到了这个问题（1-2 段）

第一层理解：我最初的做法是…… 结果…… （踩坑）

转折：后来发现真正的原因是…… （认知翻转）

正确做法 + 代码 + 数据对比

延伸：这个问题背后反映的是…… （上升一层，但不要上升到哲学）

（可选）一句有态度的收尾
```

## 质量自检清单

写完后问自己：
- [ ] 删掉所有"假设"场景，换成真实经历了吗？
- [ ] 有至少 2 个具体数字吗？（不是"大约""显著"）
- [ ] 有至少 1 个踩坑故事吗？（带报错信息）
- [ ] 有没有说过一句"取决于具体需求"？（删掉）
- [ ] 匿名发 V2EX，会不会被说"AI 写的"？
- [ ] 读完这篇，读者会收藏吗？还是读完就关了？

## 反面教材
