# 去 AI 味：技术文章专用检查手册

> AI味的本质不是某几个词。它是一种**过度的秩序感**——每段等长、每个观点都"正确"、措辞永远得体。人写的技术文章有偏见、有吐槽、有"我也不知道为什么但就是好使"。

---

## 为什么技术文章也会有 AI 味

技术文章的 AI 味和小说不同。小说是"情感假"，技术文章是**"经验假"**：
- 没踩过坑的人写不出踩坑的细节
- AI 给的方案永远"合理"，但真实开发中经常选一个"不那么合理但能跑"的方案
- AI 的技术观点没有立场，什么都"各有优劣"——真正有经验的人会说"别用 X，垃圾"

---

## 第一层：句式级

### 禁用句式（出现即改）

```
❌ "不是A，而是B" / "与其说A，不如说B" / "这不仅仅是A，更是B"
❌ "某种意义上来说" / "在这个A的时代" / "值得一提的是"
❌ "不得不说" / "毫无疑问" / "A，而非B"
❌ "事实上"作为段落开头
❌ "然而"作为段落开头（全文最多2次）
❌ "某种程度上" / "从某种角度来看"
❌ "让我们来看看" / "接下来我们将探讨"
❌ "总的来说" / "综上所述"（结尾总结八股）
❌ "希望这篇文章对你有所帮助"（AI收尾标配）
❌ "首先……其次……最后……"（三段论八股）
```

### 技术文章特有的 AI 句式

```
❌ "X 是一种强大的工具/框架/技术"
❌ "X 为开发者提供了……"（官方文档腔）
❌ "在实际开发中，我们通常会……"（伪经验，没有具体场景）
❌ "X 有以下几个优点：1. 2. 3."（产品说明书）
❌ "选择 X 还是 Y 取决于具体需求"（万能废话）
❌ "随着 X 的不断发展……"（万能开头）
❌ "X 的核心理念是……"（百科腔）
❌ "总结出一个核心实践/方法论/最佳实践"（培训PPT腔）
```

### 隐蔽的 AI 排版指纹

这些不是句式问题，是**排版习惯**暴露了 AI：

**1. 括号注释排比**
```
❌ "炸了能回滚（git revert），review 容易（每个 commit 只做一件事），中断不怕（tasks.md 写着做到哪了）"
```
三个"XX（解释）"并排，太对称了。人不会这么整齐地给三个点都加括号注释。
```
✅ 先说踩坑故事，再自然带出解决方案。用叙事引出好处，别列举好处。
```

**2. 加粗金句感**
```
❌ **所以"规"的第一原则：把所有上下文写进文件，永远不依赖对话记忆。**
❌ **计划本身也是交付物。**
❌ **上下文在文件里，不在对话里。**
```
AI 爱用加粗来"划重点"，制造金句感。一篇文章里超过 3 个加粗独句段，就开始像 AI 在做 PPT。真人写博客偶尔加粗强调，但不会每隔两段就来一句加粗的"人生感悟"。

**治法**：加粗只用于技术关键词（命令名、参数名、工具名），不用于"升华"观点。

**3. 培训PPT式引入**
```
❌ "我做 XX 和 YY 的时候，总结出一个核心实践：**每个功能开干之前，先生成 tasks.md。**"
```
"总结出一个核心实践" + 加粗金句，双重 AI 指纹叠加。
```
✅ "我做 XX 的时候养成了一个习惯——开干之前先出个 tasks.md。说白了就是个 todo list。"
```
"养成了一个习惯"比"总结出核心实践"低一档，"说白了就是"把正式感拉下来。

## 第二层：段落/结构级

### 1. 结构太工整 = AI

AI 写技术文章特别爱用完美的层级结构：
```
## 什么是 X
## 为什么用 X
## 怎么用 X
## X 的最佳实践
## 总结
```

真人写的技术文章结构是**跟着思路走的**，经常跑偏再绕回来：
- 讲着 Docker 突然吐槽了一段网络配置的坑
- 写到一半说"对了这里有个坑"然后岔开一大段
- 同一个点从不同角度说了两遍（因为第一遍没说清楚）

### 2. 每个观点都有"平衡"的对比 = AI

AI 总要两面看问题：
```
❌ "X 的优点是……但同时也存在一些局限性，比如……"
❌ "虽然 X 在某些场景下表现优异，但在另一些场景下可能不是最佳选择"
```

真人有立场：
```
✅ "别用 X。我踩过三次坑，每次都想骂人。"
✅ "X 牛逼。就这样。你要非要问缺点，那就是文档写得像屎。"
```

### 3. 代码示例太"教科书" = AI

AI 的代码示例变量名永远规规矩矩（`user`, `result`, `data`），注释齐全，没有 TODO，没有 hack。

真人的代码：
- 有 `// FIXME: 这里不知道为什么加了就好使了`
- 有 `tmp_fix_xxx` 这种变量名
- 有被注释掉的旧方案
- 注释有时候和代码对不上（改了代码忘改注释）

## 第三层：内容级

### 1. 缺乏具体踩坑细节

AI 写"踩坑经验"只能编出泛泛的坑：
```
❌ "在部署过程中遇到了一些环境配置的问题，经过排查发现是版本不兼容"
```

真人写踩坑有具体到版本号、报错信息、排查过程：
```
✅ "MySQL 8 的镜像死活连不上，报 caching_sha2_password。
   折腾了俩小时才发现是 Docker Desktop 的代理把 3306 给劫持了。
   关了 GUI 里那个 HTTP proxy 就好了。操。"
```

### 2. 没有时间线和情绪波动

AI 的叙述是平铺直叙的技术说明。真人写项目经历有时间感：
```
✅ "周五下午五点发现这个 bug。周五晚上十一点还没修好。
   周六凌晨两点，我把整个模块重写了。
   周一 code review，同事说'你这个方案不如原来的优雅'。
   我微笑着说'我知道'。"
```

### 3. 引用和数据太精确

AI 给数据总是"约 XX%"或编一个精确数字。真人经常：
- "大概快了一倍？我没细测"
- "之前那个方案跑了三四秒，改完秒出"
- "具体数字忘了，反正提升挺明显的"

## 第四层：语气级

### 技术文章的"人味"怎么来

- **口语化插入**："就……怎么说呢""反正就那么个意思""说白了就是"
- **自嘲**："我当时也觉得离谱""踩了这个坑之后我悟了""别问我怎么知道的"
- **跑题再绕回**："扯远了，说回正题""对了这里插一句"
- **对工具/框架有情绪**："webpack 配置是人能看懂的吗""Kubernetes YAML 写到怀疑人生"
- **承认不懂**："这块我其实也没完全搞明白，但目前这个方案能跑"
- **偏执**：对某个技术点反复强调，不厌其烦。"我再说一遍，一定要加这个参数"

### 避免的语气

- **讲师腔**："下面我来为大家介绍……"
- **官方腔**："该框架提供了丰富的……"
- **鸡汤腔**："编程不仅是一门技术，更是一种思维方式"
- **谦虚腔**："个人浅见，仅供参考"（知乎上这么写反而像 AI）

## 终极检测流程

### A. 句式扫描
- 搜索禁用句式列表 → 出现即改
- 数"的"字 → 连续 2 个以上重写该句
- "然而/但是/不过/事实上" → 全文不超过 3 个
- 括号注释排比 → 连续 3 个以上 `XX（解释）` 并列必须打散
- 加粗独句段 → 全文不超过 3 个，且不能用于"升华"观点

### B. 结构扫描
- 是不是完美的"总分总"？→ 打乱，允许结构不完美
- 每个小节长度是不是很均匀？→ 有的长有的短
- 有没有跑题/插嘴/吐槽？→ 没有就加
- 有没有"第一步→第二步→第三步"式的编号标题？→ 去掉编号，用自然过渡

### C. 内容扫描
- 所有技术观点是不是都"正确且平衡"？→ 加一个偏执/极端的观点
- 踩坑经验有没有具体到版本号/报错信息？→ 没有就补
- 有没有"我也不确定但就是好使"这种诚实表达？→ 没有就加

### D. V2EX/掘金 测试
随机抽 3 段，想象匿名发在 V2EX/掘金。
会不会有人回复"又是 AI 生成的洗稿"？
如果有 1% 的可能 → 那段必须改。
